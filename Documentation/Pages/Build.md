@page page_build SC::Build

`SC::Build` is a [Tool](@ref page_tools) implementing a build system generator where builds are imperatively described in C++.  

[TOC]

# Features

- Describe builds in a `.cpp` file
- Integrates with `SC-Package` from [Tools](@ref page_tools) to download dependencies
- Generate XCode 14+ Projects
    - Generate app bundle icon and storyboard
    - Link locally defined `LLDB` debug visualizer
- Generate Visual Studio 2022 Projects
    - Link locally defined `.natvis` debug visualizer
- Generate Makefile for Linux and Apple (macOS / iOS) targets
    - Generates `compile_commands.json` for VSCode
- Run a standalone native backend on Apple and Linux
    - Launch compiler and archiver processes directly without generating project files
    - Track header dependencies through compiler-generated depfiles
    - Build console executables and static libraries with workspace-local dependency ordering
    - Emit `compile_commands.json` from the native planner

# Status
🟨 MVP  
Build is currently used to generate and build the test and example projects of this repository.  
The generated-project backends remain available, and a standalone native backend is now available for direct builds on macOS, Linux and Windows.  

# Description

Build C++ files (named by convention `SC-build.cpp`) are compiled on the fly and can either generate project files for existing build systems or run the standalone native backend directly.  
The native backend invokes compiler, linker, and archiver child processes itself to produce object files, static libraries, and executables.
Projects are generated by invoking `./SC.sh build` or `SC.bat build`.

For host executables using the [Plugin](@ref library_plugin) library, `SC::Build::Project::addExportLibraries` adds the
needed `SC_EXPORT_LIBRARY_<LIBRARY>=1` defines to the executable build and also enables `-rdynamic` on Linux.

@note Check the [Tools](@ref page_tools) page for more details on `SC.sh build`.

# Native Backend

The standalone backend is selected through `SC::Build::Generator::Native`.

Current supported scope:
- Host platforms: macOS and Linux
- Toolchains: Host default, Clang-like drivers, GCC-like drivers, and `CustomDriver`
- Target kinds: console executables and static libraries
- Dependency tracking: compiler-generated depfiles
- Output layout: the same `_Build/_Outputs` and `_Build/_Intermediates` structure used by the generated backends

Typical native commands on macOS or Linux:

```bash
./SC.sh build compile SCTest Debug native
./SC.sh build run SCBuildTest Debug native
```

This is for example the `Tools/SC-build.cpp` file for the `SCTest` test suite and the [SCExample](@ref page_examples):
\include Tools/SC-build.cpp

# How to Debug

1. Make sure to compile `SC::Build` invoking any of the `Build SCTest Debug | Release` vscode tasks for your architecture
2. Launch the `Debug SC-build` for your platform from vscode

With any other debugger or IDE other than vscode, after 1. just debug `_Build/_Tools/Windows/SC-build.exe` (or the equivalent on other platform) with similar arguments as passed by the vscode task (replace `${workspaceFolder}` with the path where Sane C++ Libraries are located).
The first argument is the path to SC Libraries, then you have the tools directory and the directory of the build outputs.
As forth argument you have the name of the tool (`build` for `SC-Build`) and then any custom argument to the tool (`configure`, `SCTest` in the example below)

```json
"name": "Debug SC-build [linux] (lldb)",
"type": "lldb",
"request": "launch",
"program": "${workspaceFolder}/_Build/_Tools/Linux/SC-build",
"args": [
    "${workspaceFolder}",
    "${workspaceFolder}/Tools",
    "${workspaceFolder}/_Build",
    "build",
    "configure",
    "SCTest",
    "Debug"
],
```

# Architecture

- User adds a `SC-build.cpp` files to his library
- `SC-build.cpp` describes the SC::Build::Definition using imperative C++ code
    - Define at least one SC::Build::Workspace
    - Define at least one SC::Build::Project 
    - Define at least one SC::Build::Configuration
- `SC-build.cpp` is compiled to an executable with a single `clang` or `cl.exe` invocation
- The unity build file `SC.cpp` is linked together in the same invocation
    - Provides all needed Platform abstraction features to enumerate files and directories, launch processes etc.
- The generated executable is run as specified in SC::Build::Parameters
- `SC-build.cpp` calls SC::Build::ConfigurePresets::generateAllPlatforms or manually invokes SC::Build::Definition::generate
- Debugging the build script means just debugging a regular C++ executable

@note Check the [Tools](@ref page_tools) page for more details on `SC.sh build`.

So far the entire build configuration is created in C++ but each invocation with a different set of "build parameters" it's building a data structure that is free of conditionals, as they've been evaluated by the imperative code.
Such "post-configure" build settings could be serialized to JSON (or using binary [Serialization](@ref library_serialization_binary)) or to any other declarative format if needed.  

# Roadmap

🟩 Usable Features:
- Remove any left-over hardcoded setting
- Describe more project types (dynamic libraries, static libraries etc.)
- Describe dependencies between targets
- Allow nested builds (where the root script should compile child scripts found down the path)
- Allow different compile flags per each single file

🟦 Complete Features:
- `ninja` backend
- Extend with additional target kinds
- Generate ready to debug projects for the build program itself
- Describe very complex builds (like `LLVM`)
- Add a proper standalone cache/artifact store for native builds

💡 Unplanned Features:
- Compile scripts to WASM so that they can't run arbitrary code
